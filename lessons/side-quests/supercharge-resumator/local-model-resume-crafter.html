Local Model Resume Crafter

Estimated time: 5–6 hours | Difficulty: Advanced

Side Quest

What You Will Build

  • Use the local coding assistant from the Your Own Coding Assistant side quest as your dev-time pair programmer
  • Wire your Spring Boot Resumator to the same local Ollama model as a runtime inference engine
  • Build a Resume & Cover Letter Crafter that takes a base resume and a job description and produces a tailored, keyword-density-aware output
  • Gather a handful of resume templates as style references and let users bring their own
  • Learn prompt engineering patterns specific to keyword-density optimization and ATS-friendly output
  • Practice iterative feature development with an AI pair programmer — writing code, reviewing diffs, and keeping the human in the loop

Deliverable

A working "Craft from Scratch" button on your Resumator that, given a base resume and a job description, produces (a) a tailored resume and (b) a matching cover letter, both keyword-density-optimized for the target role, with reference-template previews and a download button. The feature runs entirely on your own hardware — no cloud API, no API key, no cost.

1. Why This Is Different from the Resume Tailor Lesson

If you completed the Resume Tailor side quest, you already built a feature that uses an AI model to tailor an existing resume. You might reasonably ask what this lesson adds. Three things:

  • Resume Tailor used a cloud LLM (OpenAI, Anthropic, Google). This lesson uses your local model — the same one you built in the Your Own Coding Assistant side quest. Your resume never leaves your machine.
  • Resume Tailor produced tailoring suggestions for an existing resume. This lesson produces a full tailored resume and a matching cover letter, both crafted from scratch against a base profile and a target job description, with explicit keyword-density targeting.
  • Resume Tailor was about building a feature. This lesson is about building a feature with your local coding assistant as your pair programmer. Every code sample in this lesson is something you will prompt your assistant to help you write, review, and refine. The meta-loop is the point.

You can do this side quest without having done Resume Tailor first, but you should have completed Your Own Coding Assistant, because the entire lesson assumes you have Ollama running locally and can talk to a model at http://localhost:11434.

Before you start: confirm your local model is running. Open a terminal and run ollama list — you should see at least one model in the list. If you pulled llama3.2:3b or qwen2.5-coder:7b during the previous side quest, you are ready. If not, run ollama pull qwen2.5-coder:7b now and come back when it finishes downloading. A coder-tuned model will produce noticeably better resume output than a general chat model.

2. The Feature, End to End

Here is what the user will see when you are done. On every job card in the Resumator, there will be a new button labeled Craft from Scratch. Clicking it opens a modal with three panels:

  1. Left panel — Inputs. A text area pre-filled with the job description from the job card, a second text area with the user's base resume loaded from their saved profile, and a dropdown to pick a reference template style.
  2. Middle panel — Generation. A "Craft" button that kicks off the pipeline. Below it, a live keyword-density meter that updates as the output is generated. Below that, a streaming preview of the resume and cover letter as they are produced.
  3. Right panel — Output. Two tabs: Resume and Cover Letter. Each has an editable preview and a download button for the final document.

Under the hood, the pipeline has four stages:

  1. Keyword extraction — parse the job description and produce a weighted list of the terms that should appear in the resume.
  2. Density targeting — decide how often each keyword should appear, based on the job title and the weight from stage 1.
  3. Resume crafting — generate a tailored resume from the base profile, hitting the keyword targets without sounding robotic.
  4. Cover letter — generate a cover letter that matches the resume's tone and reinforces the same keywords.

Each stage is a separate call to the local model. That matters: you want small, focused prompts rather than one giant prompt that tries to do everything at once. Smaller prompts give you better output, easier debugging, and faster iteration.

3. Setting Up Your Dev-Time Pair Programmer

Before you write a single line of the crafter feature, set up your local coding assistant to help you build it. This is the meta-loop: you will use the same Ollama instance for both roles, but with a different system prompt for each.

Open a new terminal. In it, start your coding assistant with a system prompt targeted at Spring Boot pair programming:

cat > ~/.coders-farm-pair-prompt << 'EOF'
You are a pair programmer for a Java 21 / Spring Boot 3 project
called Resumator. The project uses:

- Spring Boot 3 with spring-boot-starter-web and spring-boot-starter-data-jpa
- Thymeleaf for server-rendered templates
- PostgreSQL via Spring Data JPA
- Plain HTML, CSS, and vanilla JavaScript on the frontend
- Ollama at http://localhost:11434 as a local LLM backend

Rules:
- When I ask for code, produce a complete, compilable file or a
  precise, minimal diff. Never half-finish an implementation.
- Explain your reasoning in one or two sentences before the code.
- If I ask for a change that would break something else, tell me.
- If I ask something ambiguous, ask one clarifying question
  before producing code. Not five questions. One.
- Do not invent Spring Boot APIs. If you are not sure a method
  or annotation exists, say so.
EOF

Now start an interactive session with your model using that system prompt. The exact command depends on how you interact with Ollama — if you built the chat interface from the previous lesson, point it at this prompt. If you prefer raw curl, the pattern looks like this:

curl -s http://localhost:11434/api/chat -d '{
  "model": "qwen2.5-coder:7b",
  "messages": [
    {"role": "system", "content": "'"$(cat ~/.coders-farm-pair-prompt | sed "s/\"/\\\\\"/g" | tr -d "\n")"'"},
    {"role": "user", "content": "We are starting a new Resumator feature called Craft from Scratch. Before we write any code, ask me one question about the design."}
  ],
  "stream": false
}'

That is rough around the edges, and it is supposed to be — the real point is to get comfortable sending structured prompts to your local model programmatically. In practice, most students build a tiny helper script (pair.sh or pair.py) that wraps this call and keeps a conversation history. Your first task with your pair programmer is to help you write that helper. Ask it:

> Write a small Python script called pair.py that maintains a
> conversation with a local Ollama model. It should:
> - Read a system prompt from a file given on the command line
> - Loop reading user input from stdin
> - Send each user message to /api/chat with the full history
> - Stream the response back to the terminal
> - Save the full conversation to a .json file on exit

Review the output. Read every line. Run it. Fix the first thing that is wrong — there will be something, and diagnosing small bugs in AI-generated code is exactly the skill this lesson is designed to build. You now have your pair programmer. Keep it open in a second terminal for the rest of the lesson. Whenever a code sample below says "ask your assistant to help you," this is the terminal you use.

4. Gather Your Reference Templates

A good resume is not just about words — structure, layout, and tone matter enormously, and they vary by industry. Before you build the crafter, gather three to five resume templates that represent different styles. These are for the model to reference, not to copy verbatim.

Find templates that cover different looks:

  • Classic single-column — conservative, text-heavy, used in finance and academia
  • Two-column modern — skills sidebar on the left, experience on the right, common in design and product roles
  • Minimal engineering — tight, bullet-heavy, dense, used for technical roles at every level
  • Results-driven executive — numbers and impact statements front and center, common for senior positions

You can find free templates at sites like overleaf.com (LaTeX templates for engineering roles), canva.com (design-oriented templates), and the built-in template gallery in Google Docs. Do not grab ten templates — grab three or four. The model works better when the choices are clear.

For each template, save two files in a new folder src/main/resources/resume-templates/:

  • <slug>.json — metadata: name, description, style notes, target industries
  • <slug>.md — the template content as markdown, with clearly marked sections (header, summary, experience, skills, education) that the model can fill in

Here is a sample metadata file:

{
  "slug": "minimal-engineering",
  "name": "Minimal Engineering",
  "description": "Single-column, bullet-heavy, tight spacing. Optimized for technical roles at any level.",
  "styleNotes": [
    "Use strong action verbs to start every bullet",
    "Keep bullets to one line where possible",
    "Lead with technologies, follow with impact",
    "Summary is 2-3 sentences, not a paragraph"
  ],
  "targetIndustries": ["software", "devops", "data", "embedded"]
}

And here is a sample template markdown file, with placeholder markers the model can replace:

# {{name}}
{{email}} | {{phone}} | {{location}} | {{links}}

## Summary
{{summary}}

## Experience
{{experience}}

## Skills
{{skills}}

## Education
{{education}}
Encourage your users to bring their own. The folder structure above is deliberately simple so that users of your Resumator can drop their own templates in and have them show up in the picker. Part of the polish of this feature is making it extensible without requiring a code change. When you write the template-loading logic, make it scan the folder at startup and pick up anything that follows the naming convention. Ask your pair programmer for help with the @PostConstruct method that does the discovery.

5. Build the Base Resume Input

The base resume is the raw material the crafter works from. It should include everything — every job, every project, every skill, every certification — because the crafter's job is to pick the relevant parts and leave the rest out. If the base resume is pruned, the crafter has nothing to work with.

Create a new entity called BaseResume. Ask your pair programmer to help you scaffold it:

> Create a Spring Data JPA entity for a BaseResume with fields for:
> user_id (foreign key to the existing users table),
> full_name, email, phone, location, links (JSON array of strings),
> summary (text), and three JSON columns:
> experience_items, skills_items, education_items.
>
> Each experience item should include company, title, start_date,
> end_date, location, and an array of bullet strings. Skills should
> have category and a list of skill strings. Education should have
> institution, degree, graduation_date, and notes.
>
> Produce: the entity class, a Spring Data repository interface,
> and a Flyway migration SQL file.

Your assistant will give you something close to what you need. Review every line. Check the column types. Confirm the JSON mapping annotations match what your existing Resumator uses for other JSON columns. If something is off, fix it in the prompt and regenerate — do not silently patch broken AI output, teach the assistant what was wrong so the next response is better.

Why JSON columns for experience, skills, and education? Because the local model consumes JSON much more reliably than free-form text. When you pass the base resume to the crafter, you are going to serialize this entity as JSON and paste it into the prompt — and the LLM is going to extract exactly the fields it needs. A relational schema with five more tables would be more "correct" but would make the serialization and deserialization noticeably harder for no real benefit.

After the entity is in place, build a small form in the Resumator UI that lets the user edit their base resume section by section. This is not glamorous work — it is just form fields, bullet-list editors, and a save button — but the quality of the final crafted output depends entirely on how complete the base resume is. Spend the time here.

6. Build the Crafter Service

Now the interesting part. You are going to build a Spring Boot service called ResumeCrafterService that orchestrates the four-stage pipeline: keyword extraction, density targeting, resume crafting, and cover letter generation. Each stage is a separate call to your local Ollama model.

The Ollama client

First, you need a small client class that wraps the Ollama REST API. Ask your pair programmer:

> Create a Spring @Component called OllamaClient with a single
> method: String chat(String systemPrompt, String userPrompt).
> It should POST to http://localhost:11434/api/chat with model
> "qwen2.5-coder:7b", stream=false, and a two-message conversation.
> It should return the assistant's message content as a String.
>
> Use Spring's RestClient (not the deprecated RestTemplate).
> Model name and base URL should come from application.yml with
> sensible defaults.

Once your assistant produces the client, test it with a smoke-test endpoint:

@RestController
public class SmokeTestController {
  private final OllamaClient ollama;
  public SmokeTestController(OllamaClient ollama) { this.ollama = ollama; }

  @GetMapping("/smoke/ollama")
  public String smoke() {
    return ollama.chat(
      "You are a terse assistant.",
      "Say hello and confirm you are running."
    );
  }
}

Run the Resumator, hit http://localhost:8080/smoke/ollama, and confirm you get a response from your local model. If this does not work, nothing downstream will — debug here before moving on.

Stage 1: keyword extraction

Now build the first stage of the pipeline. The goal is to take a job description (free text) and produce a weighted list of keywords that should appear in the tailored resume. Here is the method signature:

public List<WeightedKeyword> extractKeywords(String jobDescription) { ... }

public record WeightedKeyword(String term, double weight, String category) {}

And here is a prompt that works well for this stage. Add it as a method in ResumeCrafterService:

private static final String KEYWORD_SYSTEM_PROMPT = """
    You are an ATS keyword extraction tool. Given a job description,
    you identify the terms that an Applicant Tracking System will
    scan for, and you assign each term a weight from 1.0 (critical)
    to 0.3 (nice to have) and a category (skill, tool, concept, soft).

    Output format: strict JSON with a "keywords" array. Each item
    has "term", "weight", and "category". Do not include anything
    except the JSON object. No commentary, no markdown fences.
    """;

private static final String KEYWORD_USER_TEMPLATE = """
    Job description:
    ---
    %s
    ---

    Extract 10 to 20 keywords. Focus on terms that appear multiple
    times or are marked as requirements. Output only the JSON object.
    """;

Notice three things about this prompt. First, it is extremely specific about the output format — strict JSON, a known structure, no markdown. Second, it gives the model a role ("you are an ATS keyword extraction tool") which narrows its behavior. Third, it caps the output (10 to 20 keywords) so the model does not run away. These three patterns will come up in every prompt in this lesson.

The method itself is short:

public List<WeightedKeyword> extractKeywords(String jobDescription) {
  String userPrompt = KEYWORD_USER_TEMPLATE.formatted(jobDescription);
  String json = ollama.chat(KEYWORD_SYSTEM_PROMPT, userPrompt);
  return objectMapper.readValue(json, KeywordResponse.class).keywords();
}

private record KeywordResponse(List<WeightedKeyword> keywords) {}

Write a small test that passes in a real job description and asserts the response is parseable and has the expected shape. Run it. If the model occasionally wraps its output in markdown code fences (it will), add a preprocessing step that strips them before parsing. Your pair programmer can help you write that cleanup logic if you ask.

Stage 2: density targeting

The next stage takes the weighted keyword list and produces a target density for each term — roughly, how many times the term should appear in the tailored resume. A keyword with weight 1.0 might have a target of 3 or 4 mentions; a weight-0.3 keyword might have a target of 1. This is largely arithmetic, not AI work — the model does not need to be involved.

public Map<String, Integer> computeTargetDensities(
    List<WeightedKeyword> keywords, int targetWordCount) {
  Map<String, Integer> targets = new LinkedHashMap<>();
  for (WeightedKeyword k : keywords) {
    int target = (int) Math.round(k.weight() * targetWordCount * 0.012);
    targets.put(k.term(), Math.max(1, target));
  }
  return targets;
}

The 0.012 magic number means "each weight-1.0 keyword should appear roughly 1.2% of the time in a 500-word resume" — about 6 mentions. Tune this number once you see real output. Keeping this stage deterministic (no LLM call) is deliberate: when something goes wrong later you will be grateful that at least one step in the pipeline is not a black box.

Stage 3: craft the resume

This is the stage where the LLM actually does the creative work. The prompt is longer than stages 1 and 2 combined, because it has to pack in everything the model needs: the base resume, the target density map, the reference template notes, and strict constraints on tone and format.

private static final String CRAFT_SYSTEM_PROMPT = """
    You are a professional resume writer. You produce tailored
    resumes that honestly represent a candidate's real experience
    while emphasizing the parts most relevant to a target role.

    Rules you MUST follow:
    - Never fabricate experience, skills, certifications, or dates.
      If something is not in the base resume, it does not appear.
    - Use the reference template's style notes for tone and layout.
    - Hit the target keyword densities as closely as possible without
      making the text sound repetitive or robotic.
    - Strong action verbs. One line per bullet where possible.
    - Do not use generic buzzwords like "synergy" or "leveraged
      cutting-edge technologies". Prefer specific, concrete language.
    - Output format: markdown matching the template's section layout.
      No commentary before or after the markdown.
    """;

private static final String CRAFT_USER_TEMPLATE = """
    Base resume (JSON):
    %s

    Target job description:
    %s

    Keyword density targets (keyword -> target mentions):
    %s

    Reference template style notes:
    %s

    Craft a tailored resume. Output only the markdown.
    """;

And the method that ties it together:

public String craftResume(
    BaseResume base,
    String jobDescription,
    Map<String, Integer> densityTargets,
    TemplateStyle template) {
  String baseJson = objectMapper.writeValueAsString(base);
  String densityText = formatDensityTargets(densityTargets);
  String styleNotes = String.join("\n- ", template.styleNotes());

  String userPrompt = CRAFT_USER_TEMPLATE.formatted(
    baseJson, jobDescription, densityText, styleNotes
  );

  return ollama.chat(CRAFT_SYSTEM_PROMPT, userPrompt);
}

This is where your local model will earn its keep. Expect the first outputs to be imperfect — maybe the density targets are not quite hit, maybe the model slips in a buzzword, maybe the formatting drifts from the template. That is fine. The next stage (keyword density verification) is how you catch and correct those issues without another human-in-the-loop round.

Stage 4: verify and iterate

After the resume is crafted, count how often each target keyword actually appears. If any are more than 30% off target, feed the actual counts back to the model and ask it to revise. Here is the pattern:

public String craftResumeWithIteration(/* ... */) {
  String draft = craftResume(base, jobDescription, targets, template);

  for (int i = 0; i < MAX_ITERATIONS; i++) {
    Map<String, Integer> actual = countKeywords(draft, targets.keySet());
    List<String> offByMore = findOffTarget(actual, targets, 0.30);

    if (offByMore.isEmpty()) return draft;

    draft = reviseDraft(draft, targets, actual, offByMore);
  }
  return draft;
}

This is a classic AI pipeline pattern: generate, measure, feed measurements back as ground truth, regenerate. The deterministic measurement step is what makes the whole pipeline trustworthy — you are not hoping the model hits the targets, you are measuring whether it did and correcting when it did not. Your pair programmer can help you write countKeywords, findOffTarget, and reviseDraft. Ask for each one individually, review, test, move on.

Stage 5: the cover letter

The cover letter is easier than the resume because it only needs to reinforce the same keywords in prose form. Feed the model the crafted resume and the job description, and ask it to produce a three-paragraph cover letter in the candidate's voice. The system prompt is shorter:

private static final String COVER_LETTER_SYSTEM_PROMPT = """
    You write professional cover letters. Three paragraphs:
    paragraph one names the role and a genuine reason for applying,
    paragraph two connects the candidate's most relevant experience
    to the job's requirements, paragraph three is a brief, confident
    closing that invites the next step.

    Rules:
    - Use the candidate's voice from the resume. Do not invent.
    - Reinforce 5 to 8 of the highest-weight keywords naturally.
    - No clichés. No "I am writing to express my interest."
    - 250 to 350 words.
    """;

The user prompt passes the crafted resume, the job description, and the top keywords. The output is plain text. Simple.

7. Build the UI

The backend is doing all the work. The UI just needs to let the user trigger the pipeline and review the output. Ask your pair programmer to help you scaffold the Thymeleaf fragment for the modal:

> I need a Thymeleaf fragment for a "Craft from Scratch" modal.
> It has three columns:
> left: textarea for job description (pre-filled from th:value),
>       textarea for base resume loaded from the user profile,
>       select for reference template (loop over ${templates}).
> middle: a "Craft" button, a div with id "density-meter" for
>         a live keyword density display, and a div with id
>         "streaming-preview" for the streaming output.
> right: two tabs (Resume, Cover Letter), each with an editable
>        contenteditable div and a Download button.
>
> Match the existing Resumator modal patterns — Bootstrap 5
> modal with our custom overrides.

Review what you get. You will almost certainly need to tweak the class names to match your existing Resumator stylesheet, and Thymeleaf's expression syntax is easy to get slightly wrong. Fix the issues, run the app, and confirm the modal opens with the job description pre-populated.

Next, write a small vanilla-JS controller for the modal. It hits your crafter endpoint with the inputs, receives the streamed output, updates the preview, and renders the density meter as each keyword appears. This is a perfect job for the Server-Sent Events (SSE) pattern you might already have in Resumator — Spring Boot makes SSE trivial with SseEmitter. Ask your pair programmer for a minimal SSE controller that streams chunks from the Ollama client to the browser as they arrive.

The keyword density meter is a nice polish moment. As the resume streams in, update a small table showing each target keyword, its target count, and its current count. Color the rows green when they hit target, yellow when they are within 30%, red when they are off. This gives the user a live sense of how the tailoring is going and doubles as a debugging aid for you.

Finally, add a download button that produces a .docx or .pdf using whichever library is easiest — Apache POI for .docx is well documented, and your pair programmer will know the boilerplate. Do not overthink the document generation; the real value of the feature is in the crafted content, not in the file format.

8. Responsible Use

Everything the Resume Tailor lesson said about responsible AI use in job applications applies here, and more so, because this feature produces entire documents from scratch rather than just suggestions. The four rules:

  1. Read every word before you send it. Your model is a 7-billion-parameter local model. It is good, but it will occasionally say something off. You are the final filter. Every sentence is something you will be expected to explain in an interview — make sure you can.
  2. Do not let it fabricate. The prompt explicitly forbids fabrication, but prompts are not contracts. The model might slip a skill you do not have into the output. If you see something in the draft that is not in your base resume, delete it.
  3. Keep the base resume honest. The crafter is only as truthful as the input. Do not inflate your base resume and expect the crafter to produce honest output. Keep the base broad but accurate.
  4. Do not mass-apply. This tool makes it easy to generate a tailored application in under a minute. Do not use that speed to fire off fifty sloppy applications. Use it to produce ten thoughtful, well-targeted ones. Quality beats quantity in every job search, and recruiters can smell mass-applied resumes from a mile away.
Why local matters here: your resume and your base profile contain personal information — your address, your phone, your salary history, your career narrative. Running the crafter on a cloud API means sending all of that to a third party with every call. Running it locally means none of it ever leaves your machine. For a feature that touches the most personal data in the Resumator, the local model is not just a cost saving — it is the right architectural choice.

Knowledge Check

1. Why does the crafter pipeline split the work into four separate model calls (keyword extraction, density targeting, resume crafting, cover letter) instead of one large prompt that does everything?

Correct. Smaller prompts give the model a single clear job at a time, which produces noticeably better output. They are also much easier to debug: if the keyword extraction is wrong you can fix that stage in isolation without touching anything else. And some steps (the density math in stage 2) do not need an LLM at all — keeping those deterministic makes the pipeline far more trustworthy, because the LLM's output can be measured and verified against ground truth.

2. What is the main architectural reason to run this feature on your local model instead of a cloud LLM API?

Exactly. Cloud models are usually faster and often higher quality, so "the local model is objectively better" is the wrong answer. The local model wins here on privacy. The base resume contains address, phone, salary history, employment gaps, and a narrative about the user's career. For a feature that handles all of that, keeping the data on the user's machine is the right architectural call even if the cloud would be slightly faster or slightly better at prose.

3. After the crafter generates a draft resume, you count each target keyword's actual occurrences and some are 40% below target. What should the pipeline do?

That is the pattern. Generate, measure, feed the measurements back as ground truth, regenerate. This is one of the most important patterns in production LLM pipelines: you never rely on the model to self-verify. You verify deterministically — counting keywords is a simple loop, not an AI task — and hand the measurements back to the model so it can correct. Capping the number of iterations prevents an infinite loop if the model cannot hit the target.
