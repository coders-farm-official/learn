Online Coding Assistants

Estimated time: 2–3 hours | Difficulty: Beginner

Side Quest

What You Will Learn

  • What an online coding assistant is, and what makes one different from a local one
  • Tour the four most popular options: Codex, Cursor, Gemini, and Claude
  • Install at least one option in VS Code or as a command-line tool — your choice
  • Understand the trade-offs between cloud assistants and the local assistant you built in the last side quest
  • Learn advanced patterns that make these tools dramatically more useful: project rules, chat vs. agent mode, inline vs. ambient, and cost control
  • Avoid the common mistakes that turn a capable assistant into a confidently wrong one

1. What Is an Online Coding Assistant?

An online coding assistant is an AI tool that helps you write, read, refactor, and debug code — with the actual model running on someone else's servers. You type a question or a request; your editor (or a terminal) packages up the relevant files and your prompt, sends them to a cloud model, and streams the reply back to you. The model never runs on your machine, but the work you do with it feels like it does.

If you finished the previous side quest, Your Own Coding Assistant, you have already built something like this — except the model lived entirely on your hardware via Ollama. Online assistants flip that trade-off. They give you access to models that are far too large to run locally (hundreds of billions of parameters) in exchange for sending your code to a third party. For most professional developers in 2026, the online option is the default. For learning, and for any project with strict privacy requirements, the local option still has an edge.

The four options we will walk through are the ones you are most likely to encounter in interviews, documentation, and on a new job:

  • Codex — OpenAI's coding assistant, built on the GPT family of models. Available as a VS Code extension and as a standalone CLI.
  • Cursor — A full editor (a fork of VS Code) with AI deeply integrated. Cursor is the editor and the assistant — you cannot really separate the two.
  • Gemini — Google's assistant, built on the Gemini family of models. Available as a VS Code extension ("Gemini Code Assist") and as a CLI.
  • Claude — Anthropic's assistant, built on the Claude family of models. Available as a VS Code extension and as a CLI called Claude Code.
Pick one or try them all. You do not need to install every option to benefit from this lesson. Read through the whole walkthrough, skip to the option that appeals to you, and get it working end-to-end. Once you have used one, the others feel familiar — the mental model carries over. If you want to try more than one, the sections are ordered so you can install each in isolation.

2. Prerequisites

Every option in this lesson expects the same short list of things already on your machine. Install these once and you are ready for any of the four walkthroughs below.

  • VS Code — free download at code.visualstudio.com. Needed for the VS Code path of every option except Cursor (which replaces VS Code). If you only plan to use the CLI path, you can skip VS Code entirely.
  • Node.js 18 or newer — download from nodejs.org. Every CLI in this lesson is distributed through npm, so you need Node to install and run them.
  • A working terminal — PowerShell on Windows, Terminal on macOS, or any Linux shell. If you finished the Linux Basics side quest, you already have everything you need.
  • Git configured with your name and email — several of these tools behave differently when they detect a Git repository. Initialize one with git init inside a practice folder before you start.
  • An account with the provider you choose — each option requires a free signup. You will create API keys or log in with a browser during the walkthrough.

Create a new folder called assistant-playground and open it in your terminal. You will use this folder as the test bed for whichever tool(s) you install.

mkdir assistant-playground
cd assistant-playground
git init
echo "# Coding assistant playground" > README.md

3. The Four Options at a Glance

Before you install anything, here is the one-screen summary. Skim it first, then skip to whichever option you want to try.

Option VS Code path CLI path Known for
Codex OpenAI extension in VS Code Marketplace npm install -g @openai/codex Strong on general problem-solving, enormous ecosystem, familiar if you already use ChatGPT
Cursor Cursor is the editor (download from cursor.com) Separate cursor-agent CLI Tightest editor integration, multi-file edits, inline "Composer" feel
Gemini "Gemini Code Assist" in VS Code Marketplace npm install -g @google/gemini-cli Massive context windows, strong Google Cloud integration, generous free tier
Claude "Claude Code" extension in VS Code Marketplace npm install -g @anthropic-ai/claude-code Long-running agentic tasks, careful reasoning, strong at following detailed instructions

Every option offers both a chat-style interaction and an agent mode that can read files, run commands, and make edits on its own. The differences show up in how they feel to use. You will get much more out of any of them by spending an hour with it than you will by reading comparison tables.

4. Option A — Codex (OpenAI)

Codex is OpenAI's coding assistant. It is built on the same GPT family that powers ChatGPT, but with tool use, file editing, and command execution layered on top. If you are already comfortable with ChatGPT's conversational style, Codex will feel instantly familiar.

VS Code installation (Codex)

  1. Open VS Code and go to the Extensions panel (Ctrl+Shift+X on Windows/Linux, Cmd+Shift+X on macOS).
  2. Search for OpenAI Codex and install the official extension (publisher: OpenAI).
  3. Click the Codex icon in the activity bar on the left side of VS Code.
  4. Sign in with your OpenAI account when prompted. You will be redirected to a browser, confirm the login, and the extension will pick up the session automatically.
  5. Open your assistant-playground folder in VS Code (File → Open Folder). The Codex panel now knows the contents of that folder and can reference files in its replies.

CLI installation (Codex)

  1. Install the Codex CLI globally with npm: 
    npm install -g @openai/codex
  2. Run codex inside your assistant-playground folder. On first run it will ask you to sign in — either through a browser redirect or by pasting an API key from platform.openai.com/api-keys.
  3. You will land in an interactive prompt. Try asking it to create a file: 
    > Write a Python script called hello.py that greets the user by name.
  4. Codex will propose a diff, ask you to confirm, and then write the file. Review the change before you approve it — this is a habit you will want from day one.

Codex strengths and drawbacks

  • Strengths: familiar conversational style, strong general knowledge, excellent at explaining concepts, huge community around prompt patterns, works well for "explain this code" kind of questions.
  • Drawbacks: agent mode is powerful but can be aggressive — it will happily rewrite half your codebase if you let it. Always run inside a Git repo so you can git diff and revert.

5. Option B — Cursor

Cursor is not a VS Code extension — it is a whole separate application, a fork of VS Code with AI built into the core of the editor. When you install Cursor you are installing an editor that looks and feels almost identical to VS Code, but the AI is woven into every part of the interface: tab completion, inline edits, a side panel, and a multi-file "Composer" mode.

Editor installation (Cursor)

  1. Download Cursor for your platform from cursor.com. Windows, macOS, and Linux are all supported.
  2. Run the installer and launch Cursor.
  3. On first launch, Cursor will ask if you want to import your VS Code settings and extensions. Say yes — it will pick up your themes, keybindings, and installed extensions automatically.
  4. Sign in with a Cursor account (free tier available) when prompted. This links your editor to the cloud model.
  5. Open your assistant-playground folder. Press Ctrl+L (or Cmd+L on macOS) to open the chat panel, or Ctrl+K for an inline edit prompt on the current file.

CLI installation (Cursor)

Cursor also ships a separate command-line tool called cursor-agent that gives you headless access to the same model from a terminal. Installation depends on your platform — check the official documentation at cursor.com/cli for the current install command, as the recommended method changes occasionally. At the time of writing, the most reliable path on macOS and Linux is a one-line installer script; on Windows, the installer is a standalone executable.

Once installed, run cursor-agent inside your playground folder to start an interactive session backed by the same model the editor uses. The CLI is useful for scripted workflows — for example, running a prompt across every file in a directory as part of a build step.

Cursor strengths and drawbacks

  • Strengths: by far the tightest editor integration of any option. Multi-file edits via Composer feel magical the first few times. Tab-completion is aggressive and usually right. No separate "chat and then apply" dance — the AI lives inside the editor surface.
  • Drawbacks: you have to use a different editor. If your team is committed to plain VS Code, Cursor creates friction. The free tier has a monthly usage cap that serious users will hit quickly; a paid plan is effectively required for daily use.

6. Option C — Gemini (Google)

Gemini is Google's coding assistant, built on the Gemini family of models. Its headline feature is an enormous context window — Gemini can keep an entire medium-sized codebase in view at once, which changes how you prompt it. If you have ever wanted to ask "which files in this project handle authentication?" and get a good answer, Gemini's context window makes that realistic.

VS Code installation (Gemini)

  1. Open VS Code and go to the Extensions panel.
  2. Search for Gemini Code Assist and install the extension published by Google.
  3. Click the Gemini icon in the activity bar. The extension will prompt you to sign in with a Google account.
  4. During sign-in, Gemini Code Assist will also ask you to enable a Google Cloud project. You can use a free personal project — no billing card required for the free tier.
  5. Open your playground folder. Use the Gemini panel to ask questions, or press the keyboard shortcut shown in the extension description for inline edits.

CLI installation (Gemini)

  1. Install the Gemini CLI with npm: 
    npm install -g @google/gemini-cli
  2. Run gemini inside your playground folder. It will walk you through authentication — either a browser-based Google sign-in or pasting an API key from aistudio.google.com/apikey.
  3. Try a prompt that takes advantage of the large context window: 
    > Read every file in this folder and tell me what the project does.
  4. Gemini will read everything, summarize, and then wait for your next instruction. This is the pattern that feels most different from Codex or Claude — Gemini expects to see a lot of your code at once.

Gemini strengths and drawbacks

  • Strengths: the context window is the headline feature and it genuinely matters — you can paste a whole file without worrying about truncation. Generous free tier. Tight integration with Google Cloud if you work in that ecosystem.
  • Drawbacks: the UX has more rough edges than Codex or Cursor — sign-in and project setup can be confusing the first time. When a model has the whole codebase in view, it is also more likely to make sweeping changes you did not ask for. Review diffs carefully.

7. Option D — Claude (Anthropic)

Claude is Anthropic's coding assistant, and its public product for developers is called Claude Code. Claude is known for careful, deliberate responses — it tends to ask clarifying questions, point out trade-offs, and follow detailed instructions closely. In exchange, it can feel slower than Codex on simple questions. For longer, multi-step tasks ("refactor this service to use constructor injection instead of field injection") Claude tends to shine.

VS Code installation (Claude)

  1. Open the Extensions panel in VS Code.
  2. Search for Claude Code and install the extension published by Anthropic.
  3. Click the Claude icon in the activity bar and sign in with your Anthropic account. A browser window will open, you confirm, and the extension picks up the session.
  4. Open your playground folder. You can now chat with Claude, ask it to edit files, or hand it a longer task and watch it work through the steps.

CLI installation (Claude)

  1. Install Claude Code globally: 
    npm install -g @anthropic-ai/claude-code
  2. Run claude inside your playground folder. The CLI will open a browser tab for sign-in on first run, or you can paste an API key from console.anthropic.com.
  3. Try an agentic task: 
    > Create a small Express server in server.js that returns "Hello, Coders Farm!" on GET /. Then run it and tell me the port.
  4. Claude will propose the file, ask to run npm init -y and install express, wait for you to approve, and then start the server. You confirm each step — that permission-on-every-command rhythm is the defining feel of the Claude Code CLI.

Claude strengths and drawbacks

  • Strengths: the strongest option for longer multi-step tasks where you want the assistant to think carefully and ask permission before acting. Strong at reading unfamiliar codebases. Excellent at following a detailed system prompt exactly. The permission-per-command style in the CLI is reassuring for beginners.
  • Drawbacks: can feel slow on trivial questions where you just want a quick answer. Free usage is more limited than Gemini's. If you want a "just autocomplete my code" experience, Cursor is a better fit.

8. How This Differs from Your Own Coding Assistant

If you completed the previous side quest, you already built a local coding assistant using Ollama. That assistant runs entirely on your hardware — no network calls, no accounts, no cost per token. So why would you ever use an online one?

The honest answer is that the two tools serve different purposes, and most professional developers end up using both. Here is how they compare on the things that matter:

Dimension Local (Ollama) Online (Codex/Cursor/Gemini/Claude)
Privacy Your code never leaves your machine Every prompt and every file you reference is sent to a third party
Cost Free after hardware purchase Free tier + paid plan (usually $10–$30/month for serious use)
Model size 3B–13B parameters on consumer hardware Hundreds of billions to trillions of parameters
Quality Good for familiar patterns, struggles with novel problems Strong across the board, handles unfamiliar codebases well
Speed Limited by your hardware — can be very slow on large prompts Fast, even on massive prompts
Works offline Yes — no internet required No — cloud connection is mandatory
Works inside a locked-down corporate network Yes Usually blocked unless IT approves the provider
Teaching value Huge — you see exactly how the model fits together Black box — you see the output but not the machinery

A reasonable real-world pattern is: use the online assistant for day-to-day work, and keep a local model around for privacy-sensitive tasks and for learning. Running your own model taught you how LLMs actually work. Using an online model teaches you how professional development teams use AI in 2026. Both skills matter.

There is one more category worth mentioning. Some organizations run a self-hosted enterprise model — a large model running on their own GPUs, accessed through an API that looks like the online options but with all traffic staying inside the company network. If you ever work at a bank, a hospital, or a government agency, this is what you will probably use. It has the privacy of local with the scale of online. From your perspective as a developer, it behaves almost exactly like one of the online assistants — the same prompt patterns, the same mental model, different endpoint.

9. Advanced Usage

You can get value from any of these tools in the first five minutes by typing questions into the chat panel. But the difference between an average user and a power user is huge — and it comes down to a handful of patterns that work across every option. Learn these and your output quality will jump noticeably.

Give it a project rules file

Every option supports some form of project rules — a plain-text file at the top of your repository that the assistant reads automatically on every interaction. This file is where you put the things you would otherwise have to repeat in every prompt: coding conventions, tech stack, testing rules, things to avoid. The file has different names for different tools:

  • Codex looks for AGENTS.md at the project root.
  • Cursor looks for .cursorrules or the newer .cursor/rules/ directory.
  • Gemini looks for GEMINI.md.
  • Claude looks for CLAUDE.md.

The content of the file is similar in every case. Here is a starter you can adapt to any of them:

# Project rules

## Stack
- Java 21, Spring Boot 3, Maven
- PostgreSQL, accessed through Spring Data JPA
- Thymeleaf for server-rendered HTML

## Conventions
- Every new endpoint must have an integration test in the same package
- Prefer constructor injection over field injection
- Never commit API keys; use environment variables loaded via application.yml
- All user input must go through a validation layer

## Things to avoid
- Do not introduce new dependencies without asking first
- Do not rewrite code that already passes tests just to make it "cleaner"
- Do not run destructive database migrations without confirmation

Drop this file at the root of your project and every future prompt gets it for free. The payoff is enormous — you stop fighting the assistant's defaults and it starts fitting into your codebase.

Know the difference between chat, inline, and agent mode

Every online assistant offers three different interaction styles, even if they call them different names. Understanding when to use which one is half the battle:

  • Chat mode — you type a question in a side panel and get a reply. Best for exploring unfamiliar code, understanding concepts, and deciding what to do. Nothing changes in your files until you copy-paste.
  • Inline edit mode — you select a chunk of code, press a shortcut (Ctrl+K in Cursor, similar in each tool), and describe a change. The assistant rewrites just that selection. Best for targeted edits you already know you want.
  • Agent mode — you give the assistant a larger task and it runs multiple steps on its own: reading files, writing files, running commands. Best for longer tasks where the steps are too tedious to do one at a time, but review every action it takes.

The biggest beginner mistake is putting agent-mode-sized tasks into chat mode and vice versa. If you catch yourself pasting the same file into a chat panel six times, you wanted inline or agent mode. If you are reviewing a 500-line diff from agent mode and only wanted to fix one typo, you wanted inline.

Prompt patterns that work everywhere

Treat prompts like small specifications. The more structure you give the assistant, the better the output. The pattern below works across every option:

Context: [what you're working on and why]
Task: [the specific thing you want done]
Constraints: [what NOT to change or assume]
Output format: [what the reply should look like]

Here is a concrete example. Instead of writing "add pagination to the jobs page," write:

Context: This is the Resumator project from the Coders Farm
curriculum. Jobs are served by JobController.java and rendered
by jobs.html with Thymeleaf. Right now all jobs load at once.

Task: Add server-side pagination with a page size of 20. Use
Spring Data's Pageable.

Constraints: Do not change the Job entity. Do not touch
JobSearchService — only the controller and the template.
Keep the existing search bar working.

Output format: Show the updated JobController.java in full,
then the diff for jobs.html, then a one-sentence summary of
what changed.

The assistant will produce dramatically better output from the second prompt than the first. This is the single biggest lever you have for getting value out of an online assistant. It costs you thirty seconds and it saves you ten minutes of back-and-forth.

Treat the assistant like a junior teammate, not an oracle

Online assistants confabulate — they occasionally produce code that looks plausible but refers to APIs that do not exist, functions with the wrong signature, or configuration keys the framework has never heard of. The models are better than they were a year ago, but the failure mode has not gone away.

The mental model that works is this: treat the assistant like a bright, fast, slightly overconfident junior teammate. You would not merge a junior's PR without reading it. You would not paste their suggestion into production without running the tests. Apply the same discipline here. Read every diff before you accept it. Run the tests. Click around the UI. If you do not understand what the assistant changed, you are not ready to keep it.

Control the cost

Online assistants bill by token usage — roughly, the number of characters in your prompt plus the characters in the reply. Every option has a free tier, but serious use will push you into a paid plan quickly. Three habits keep the cost under control:

  • Be specific about what files the assistant should read. Agent mode will happily slurp up your entire repo into the context window if you let it. Point it at the two or three files that actually matter.
  • Use the smaller, cheaper model for small tasks. Every provider offers a "mini" or "flash" model that costs a fraction of the flagship. For tab-completion and quick questions, the small model is almost always fine.
  • Check the usage dashboard weekly. Every provider has one. Look at it. Understand where your tokens go. If one kind of task is eating half your budget, that is the task to rethink.

MCP and tool use

The newest capability across every option is Model Context Protocol (MCP) support — a way for you to extend an assistant with custom tools. You can give your assistant access to your database, your build system, your issue tracker, or any other system you care about, and it will use those tools to answer your questions. MCP is a rabbit hole worth exploring once you are comfortable with the basics. For now, just know it exists and that it is the feature that separates "AI assistant" from "AI coworker."

10. Common Mistakes

Most people get mediocre results from these tools for the same small set of reasons. Watch for these in your own workflow:

  • Vague prompts. "Fix the bug" is not a prompt. "The login form returns 500 when the email contains a plus sign — I think it is URL encoding, see AuthController.java line 42" is a prompt. Specificity is free; use it.
  • Accepting diffs without reading them. The moment you stop reviewing, you start introducing bugs you did not write. You will not notice at first. You will notice in production.
  • Asking the assistant to make stylistic changes that span hundreds of files. It can do this. It will sometimes do it slightly wrong on a handful of files in ways you will not catch until weeks later. Use your IDE's refactor tools for mechanical changes; save the assistant for the judgment calls.
  • Ignoring the rules file. If you find yourself typing the same preamble into every prompt ("remember, we use Spring Boot 3 and Thymeleaf..."), stop. Put it in the rules file. The rules file is a gift to your future self.
  • Treating one assistant as "the best." They have different personalities and different strengths. Most professional developers keep two installed and switch between them depending on the task.

Knowledge Check

1. What is the most important single pattern for getting dramatically better output from any online coding assistant?

Correct! Structured prompts with context, task, constraints, and output format — combined with a project rules file for the repetitive stuff — are the single biggest lever you have. Bigger models help a little. Better prompts help a lot. Every option in this lesson (Codex, Cursor, Gemini, Claude) supports a rules file at the project root, and every option benefits equally from structured prompts.

2. When is a local coding assistant (the one you built with Ollama) still the right choice over an online option?

Exactly. The online options win on raw quality, context window size, and speed on large prompts. The local option wins on privacy (your code never leaves the machine), cost (free after the hardware), and offline capability. Hospitals, banks, and government agencies often cannot use cloud assistants at all — so the local option is not just a learning exercise, it is a real-world requirement for a lot of professional work.

3. You are using Cursor's Composer to make a large multi-file change. The assistant produces a 400-line diff. What should you do next?

That is the right habit. Treat the assistant like a bright, overconfident junior teammate: you would never merge a junior's 400-line PR without reading it, running the tests, and making sure you understood it. The same discipline applies here. The moment you stop reviewing is the moment you start introducing bugs you did not write. This habit is more important than which assistant you pick.

Deliverable

Pick one of the four options and get it working end-to-end in your assistant-playground folder. Your deliverable is:

  • The option is installed (either the VS Code path or the CLI path — your choice)
  • You have a project rules file (AGENTS.md, .cursorrules, GEMINI.md, or CLAUDE.md) committed to your playground repo
  • You used the assistant to build a small project of your choice — a single-file HTML tic-tac-toe, a Python script that fetches weather data, a short Java program, anything you like — using a structured prompt and reviewing every diff before accepting it
  • You can answer: which option did you pick, why, and what would you change about your prompt on the next iteration?

Bonus: install a second option and build the same small project with it. Compare how the two assistants approached the problem, what they got right, and where they differed. There is no "best" assistant — only a best fit for a particular task and personal style. The fastest way to develop taste is to see the same problem solved by two different tools.

You have now used a real professional-grade coding assistant. That skill carries into every job you will have from here forward. Combine it with the local assistant you built in the last side quest and you understand the full spectrum of how developers use AI in 2026 — from a 3-billion-parameter model running on a Raspberry Pi to a trillion-parameter model running in a data center. Both are tools. Both have a place. You now know when to reach for each one.
